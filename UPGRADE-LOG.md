# Employee Interview Analyzer 升级日志

## v2.12.0 (2026-03-25) - 快速分析模式 ⚡🚀

### 📌 升级目标

为简单/短访谈场景提供「快速分析模式」，通过自动内容复杂度评估减少不必要的交互轮次和分析步骤，提升用户体验和分析效率。同步进行文档模块化优化，减少 SKILL.md 体积。

---

### ⭐⭐⭐ 核心变更：内容复杂度评估引擎

**变更说明：**
在 Step 1 场景识别完成后，新增自动复杂度评估环节。评估结果决定后续流程采用标准模式还是快速模式。

**评估维度（5维度，总分0-10）：**

| 维度 | 0分 | 1分 | 2分 |
|------|-----|-----|-----|
| 字数 | ≤1500字 | 1500-5000字 | >5000字 |
| 话题数 | ≤3个 | 3-5个 | >5个 |
| 访谈人数 | 1人 | 2人 | >2人 |
| 场景复杂度 | 入职/职业发展等 | 绩效/晋升 | 组织诊断/综合分析 |
| 矛盾信号 | 0-1个 | 2-3个 | >3个 |

**触发阈值：** 总分 ≤ 3 → 自动进入快速模式

---

### ⭐⭐⭐ 核心变更：快速分析模式行为

**快速模式步骤变更表：**

| 步骤 | 标准模式 | 快速模式 |
|------|---------|---------|
| Step 4 (框架确认) | 展示并请用户确认 | ⏭️ 跳过确认，自动采用 |
| Step 6 (交叉验证) | 完整执行 | ⏭️ 简化，仅关键矛盾检测 |
| Step 9 (组织画像) | 组织诊断时执行 | ⏭️ 跳过（快速模式不含组织诊断） |
| Step 10 (可视化) | 强制询问 | ⏭️ 默认跳过不询问 |
| Step 11 (结论文章) | 强制询问 | ⏭️ 默认跳过不询问 |
| Step 13 (腾讯文档) | 强制询问 | ⏭️ 默认跳过不询问 |
| Step 3 (方法论) | 最多3种 | 限制为2种 |
| Step 12 (检查清单) | 13项全检查 | 仅检查必执行步骤 |

**安全限制：**
- 组织诊断和综合分析场景禁止进入快速模式
- Step 1/2/3/5/7/8/12 不可省略

**用户交互：**
- 触发方式A：自动触发（复杂度 ≤ 3），显示提示
- 触发方式B：用户手动请求（`"快速分析"` `"简单分析"`）
- 退出方式：用户回复 `"详细分析"` 或 `"完整模式"`

## v2.11.0 (2026-03-25) - 管理干部画像对齐HR系统维度 🏢📋

### 📌 升级目标

将管理干部画像的多场景管理表现维度与HR系统完全对齐，确保画像数据可直接录入HR系统，实现分析结果与人事系统的无缝衔接。

---

### ⭐⭐⭐ 核心变更：行为标签对齐HR系统三分类

**变更说明：**
管理干部画像中每条行为记录的表现标签从原有的三色标记体系，改为HR系统标准三分类。

| 维度 | 旧版（v2.6.0-v2.10.0） | 新版（v2.11.0） |
|------|----------------------|----------------|
| 正向表现 | 🟢 正向 | `正向行为` |
| 负面表现 | 🔴 负向 | `需关注和待提升行为` |
| 中立表现 | 🟡 中立 | `无标签` |

**对齐理由：**
- HR系统录入字段为文本标签，非emoji标记
- 「需关注和待提升行为」比「负向」更具建设性和行动导向
- 「无标签」替代「中立」，避免模糊判断

---

### ⭐⭐⭐ 核心变更：场景类型改为可多选

**变更说明：**
每条行为记录的场景分类从单选改为可多选，支持一条行为横跨多个管理场景。

| 维度 | 旧版（v2.6.0-v2.10.0） | 新版（v2.11.0） |
|------|----------------------|----------------|
| 分类方式 | 单场景分类（S1-S6） | 可多选场景类型 |
| 场景名称 | 业务管理场景、团队管理场景... | 业务管理、团队管理... |
| 可选值 | S1/S2/S3/S4/S5/S6 | 业务管理/团队管理/周边协作/资源使用/自我管理/其他 |

**去掉"场景"后缀**：如"业务管理场景"→"业务管理"，与HR系统字段名完全一致。

---

### ⭐⭐ 核心变更：行为描述STAR原则标准化

**变更说明：**
行为描述严格按 STAR 原则结构化输出：
- **S**ituation（情境）：描述行为发生的背景和环境
- **T**ask（任务）：说明面临的任务或挑战
- **A**ction（行动）：描述具体采取的行动
- **R**esult（结果/影响）：说明行动带来的结果和影响

字数控制：200-600字，350字左右最佳。

---

### ⭐⭐ 核心变更：新增发生时间字段

**变更说明：**
每条行为记录新增「发生时间」字段，从访谈内容中提取或推断行为发生的大致时间。

**提取规则：**
- 明确时间表述：直接提取（如"去年Q3"、"2025年下半年"）
- 隐含时间线索：推断并标注 `[推断]`（如"刚加入团队时"→推断为入职初期）
- 无法判断：标注 `[未知]`

---

### ⭐ 增强功能：行为记录独立编号

**变更说明：**
每条行为记录独立编号（如 B1、B2、B3），支持同一访谈提取多条行为记录，便于HR系统逐条录入和引用。

行为记录采用**表格+描述**的标准化格式，表头包含：编号、行为标签、场景类型、发生时间、STAR描述摘要。

- 修正 Step 2 文本修正缺少独立标题标记的问题，补充 `### Step 2: 文本修正 ⭐⭐`

## v2.10.0 (2026-03-25) - 强化流程执行机制 ⚡

### 📌 升级目标

强化分析流程的执行可靠性，确保Step 8画像在非离职场景强制执行，完善Step 10/11/13的询问流程，新增流程状态追踪和错误处理机制。

---

### ⭐⭐⭐ 核心变更：Step 8 强制执行机制

**变更说明：**
明确规定除离职访谈外，所有场景（包括AI使用情况访谈等特殊场景）的Step 8画像生成为强制步骤。

**画像类型自动判定逻辑：**
```
非离职场景 → 必须生成画像
  ├─ 招聘面试 → 候选人综合画像
  ├─ is_manager=True → 管理干部画像
  └─ is_manager=False → 普通员工画像
```

**新增保障机制：**
- 画像生成失败时提供重试机制和错误提示
- 在强制完成检查清单中明确画像生成状态校验
- 不可静默跳过画像生成

---

### ⭐⭐⭐ 核心变更：Step 10/11/13 强制询问流程

**变更说明：**
将Step 10/11/13从"可选"升级为"强制询问"模式。这些步骤不再允许跳过询问，必须主动询问用户后根据回应执行或跳过。

**询问执行顺序：**
1. Step 7 完成 → **立即执行** Step 8 画像生成（非离职场景强制）
2. Step 8 完成 → **询问** Step 10（数据可视化）
3. Step 10 执行/跳过 → **询问** Step 11（结论文章）
4. Step 11 执行/跳过 → **执行** Step 12（输出完整报告）
5. Step 12 完成 → **询问** Step 13（腾讯文档）

**询问状态追踪：**
- 必须记录每个询问步骤的状态（已询问/用户确认/用户拒绝/未回应）
- 确保每个询问都被正确处理，不可遗漏

---

### ⭐⭐ 新增功能：流程状态追踪机制

**变更说明：**
新增流程状态变量，在关键节点输出当前进度提示。

**进度提示格式：**

### ⭐ 新增功能：错误处理与恢复

## v2.8.0 (2026-03-25) - 三轨画像系统、候选人综合画像 🎯

### 📌 升级目标

引入三轨画像系统，根据访谈场景自动选择画像策略；新增招聘面试专用的候选人综合画像模板；方法论配置改为优先2种。

---

### ⭐⭐⭐ 核心变更：三轨画像系统（Step 8 场景化控制）

**变更说明：**
Step 8 画像生成根据 Step 1 场景识别结果自动选择画像策略：

| 访谈场景 | 画像策略 | 说明 |
|---------|---------|------|
| 离职访谈 | ❌ 跳过 Step 8 | 聚焦离职原因分析，不生成画像 |
| 招聘面试 | 候选人综合画像 | 使用专用模板，不使用员工/管理干部画像 |
| 其他场景 | 双轨画像 | 沿用 v2.7 逻辑（管理干部画像/普通员工画像） |

**判定逻辑（伪代码）：**
```python
def determine_profile_type(scene, is_manager):
    if scene == "离职访谈":
        return "skip"
    elif scene == "招聘面试":
        return "candidate_profile"
    elif is_manager:
        return "manager_profile"
    else:
        return "employee_profile"
```

---

### ⭐⭐⭐ 新增功能：候选人综合画像模板

**功能描述：**
招聘面试场景专用画像模板，结构如下：

1. **基础信息**：候选人姓名、应聘岗位、当前公司/职位、工作年限等
2. **能力优劣势分析**：核心优势和潜在劣势，均附面试原文证据
3. **发展潜力评估**：学习能力、抗压能力、成长动力等5维度评级
4. **个人发展诉求**：短期目标、长期愿景、核心诉求
5. **岗位匹配分析**：基于用户提供的岗位信息生成匹配矩阵
6. **STAR实际工作场景呈现**：按STAR原则还原候选人描述的真实工作场景
7. **综合评价**：专业能力/岗位匹配度/发展潜力/文化适配度/风险评估 + 录用建议

---

### ⭐⭐ 核心变更：方法论优先2种

**变更说明：**
默认配置从1主+2辅=3种改为1主+1辅=2种，仅当访谈内容复杂度高时才扩展至3种。

```python

```

### ⭐⭐ 新增功能一：文本自动修正（Step 0.5）

**功能描述：**

**修正范围：**
1. **错别字修正**
   - 常见同音字错误（如：的/得/地、在/再、已/以等）
   - 形近字错误
   - 输入法选词错误

2. **语句不通修正**
   - 语序混乱的句子
   - 缺少主语或宾语的句子
   - 重复冗余的表达
   - 标点符号错误

3. **上下文理解修正**
   - 基于上下文推断可能的正确用词
   - 保留口语化表达，但修正明显错误
   - 不改变原意，仅提升可读性

**修正规则示例：**

| 规则类型 | 示例 | 修正后 |
|---------|------|-------|
| 错别字 | "这个项目做的不错" | "这个项目做得不错" |
| 语句不通 | "因为很多问题，所以导致" | "因为有很多问题，所以导致" |
| 标点错误 | "我们要解决问题，然后才能前进" | "我们要解决问题，然后才能前进" |

**修正输出：**
- 所有修正记录在分析报告附录中呈现
- 包含原文、修正后、修正理由
- 确保修正过程可追溯

**技术实现：**
```python
def correct_interview_text(text: str) -> dict:
    """
    修正访谈文本中的错误
    返回: {
        "corrected_text": "修正后的文本",
        "corrections": [
            {
                "original": "原文片段",
                "corrected": "修正后片段",
                "type": "错别字/语句不通/标点错误",
                "reason": "修正理由"
            }
        ]
    }
    """
    # 1. 错别字修正
    # 2. 语句不通修正
    # 3. 标点符号修正
    # 4. 上下文一致性检查
```

- ⚠️ 记录修正：所有修正都要记录在报告中
- ✅ 提升准确性：文本修正有助于提升后续分析准确性

### ⭐⭐ 新增功能二：数据可视化自动判断

**功能描述：**
根据访谈内容自动判断是否需要生成数据可视化图表，无需询问用户。

**自动判断规则：**

| 条件 | 说明 | 触发可视化 |
|------|------|-----------|
| **多维度评分** | 访谈分析中有3个以上维度的量化评分 | ✅ 自动生成 |
| **对比分析** | 存在多个对象、多个时间点、多个维度的对比 | ✅ 自动生成 |
| **趋势分析** | 多次访谈或时间序列数据 | ✅ 自动生成 |
| **优先级评估** | 问题评分、改进优先级等 | ✅ 自动生成 |
| **简单访谈** | 单次访谈、无量化评分、无对比需求 | ❌ 不生成 |

**判断逻辑：**
```python
def should_generate_visualization(analysis_result: dict) -> dict:
    """
    判断是否需要生成数据可视化
    """
    chart_types = []
    
    # 1. 检查是否有量化评分（3个以上维度）
    if len(analysis_result.get("scores", {})) >= 3:
        chart_types.append("radar")
    
    # 2. 检查是否有对比数据
    if analysis_result.get("has_comparison"):
        chart_types.append("bar")
    
    # 3. 检查是否有时间序列
    if analysis_result.get("has_trend"):
        chart_types.append("line")
    
    # 4. 决定是否生成
    should_generate = len(chart_types) > 0
    
    return {
        "should_generate": should_generate,
        "chart_types": chart_types,
        "reason": "检测到{}类适合可视化的数据".format(len(chart_types))
    }
```

**支持的可视化类型：**
1. **雷达图**：多维度综合评估（杨三角、韦斯伯德、冰山模型等）
2. **柱状图**：维度对比分析（健康度评分、选项统计等）
3. **折线图**：趋势分析（时间序列、多次访谈对比等）

   - 适用场景：维度对比分析
   - 应用案例：健康度评分对比、推拉力强度对比、优先级矩阵
   - 技术实现：Chart.js

3. **折线图**
   - 适用场景：趋势分析
   - 应用案例：多次访谈时间序列、绩效变化趋势、成长轨迹
   - 技术实现：Chart.js

**使用流程：**
```
分析完成后，询问用户是否需要生成可视化图表：
- 1️⃣ 雷达图
- 2️⃣ 柱状图  
- 3️⃣ 折线图
- 4️⃣ 全部生成
- 5️⃣ 暂不需要
```

### 📝 新增功能三：访谈结论文章生成

**功能描述：**
基于访谈分析结果自动生成800-1500字的访谈结论文章，确保科学合理、易读性强、切合访谈事实。

**文章规范：**
- **字数范围**：800-1500字
- **科学合理**：基于访谈事实，避免主观臆断
- **易读性强**：结构清晰、语言流畅、重点突出
- **切合事实**：引用访谈内容作为支撑证据

**文章结构（标准模板）：**
```markdown
1. 访谈背景（150字）
   - 访谈时间、对象、目的
   - 访谈场景概述
   - 核心议题

2. 核心发现（400字）
   - 主要发现1 + 证据
   - 主要发现2 + 证据
   - 主要发现3 + 证据

3. 深度分析（300字）
   - 问题根因分析
   - 矛盾点分析
   - 关键洞察

4. 建议与展望（250字）
   - 短期建议
   - 长期展望
   - 后续行动

5. 总结（100字）
   - 核心结论
   - 关键要点
```

**文章类型：**
1. **组织诊断型**：适用于组织诊断访谈
2. **绩效评估型**：适用于绩效访谈
3. **离职分析型**：适用于离职访谈
4. **职业发展型**：适用于职业发展访谈
5. **自动推荐**：根据访谈内容自动推荐类型

**质量评估：**
| 维度 | 评估标准 | 满分 |
|------|---------|------|
| 科学性 | 基于事实、论证合理、方法科学 | 30 |
| 准确性 | 数据准确、引用正确、无误导性表述 | 25 |
| 易读性 | 结构清晰、语言流畅、重点突出 | 25 |
| 完整性 | 信息完整、逻辑连贯、结论明确 | 20 |
| **总分** | | **100** |

**使用流程：**
```
分析完成后，询问是否生成访谈结论文章：
- 1️⃣ 生成文章
- 2️⃣ 暂不需要
```
