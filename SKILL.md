# Employee Interview Analyzer - 员工访谈结果智能分析技能

专为互联网公司 HR 设计的**访谈结果智能分析**技能。输入访谈记录/纪要，自动识别访谈场景，智能匹配分析方法论，输出结构化洞察报告。

> **与 skill-b（employee-interview-generator 提纲生成器）的关系：**
> - **skill-b（提纲生成器）** → 访谈**前**：生成专业的访谈提纲
> - **skill-a（本技能/分析器）** → 访谈**后**：分析访谈结果，提取深度洞察
> - 两个技能共享相同的方法论体系，形成完整的**访谈前→访谈后**闭环

---

## 🎯 何时使用

**当 HR 完成员工访谈/面试后，需要对访谈记录进行专业分析时触发此技能。**

### 支持的分析场景

| 分析类型 | 触发场景 | 输入要求 |
|---------|---------|---------|
| **入职访谈分析** | 分析新员工融入情况 | 入职访谈记录 |
| **绩效访谈分析** | 分析绩效问题根因 | 绩效面谈记录 |
| **离职访谈分析** | 分析离职原因 | 离职访谈记录 |
| **晋升访谈分析** | 评估晋升准备度 | 晋升评估面谈记录 |
| **职业发展分析** | 分析职业发展路径 | 职业发展面谈记录 |
| **团队反馈分析** | 诊断团队问题 | 团队访谈/反馈记录 |
| **转岗访谈分析** | 评估转岗适配度 | 转岗面谈记录 |
| **组织诊断分析** ⭐ | 系统性组织健康诊断 | 组织诊断访谈记录（支持多人） |
| **招聘面试分析** ⭐ | 评估候选人匹配度 | 面试记录/评估表 |
| **综合分析** | 多次访谈的趋势分析 | 多份访谈记录 |

### 关键触发短语

```
- "帮我分析一下这份访谈记录"
- "这是和 XXX 做完访谈的纪要，帮我做个分析"
- "分析一下这次离职访谈"
- "帮我看看这个组织诊断的结果"
- "这几份绩效面谈记录，帮我做个综合分析"
- "分析一下这个候选人的面试表现"
- "这份访谈记录用什么方法论分析比较好？"
```

### Step 1: 智能场景识别引擎（自动执行） ⭐

分析引擎通过**关键词匹配 + 语义理解 + 上下文推断**三层机制，自动识别访谈场景类型。无需询问用户，直接基于上下文进行识别。

#### ⭐⭐ 管理干部身份识别（v2.6.0 新增）

> **核心原则：在 Step 1 场景识别的同时，必须判断访谈对象是否为带团队的管理干部。此判断结果将决定 Step 8 使用「管理干部画像」还是「普通员工画像」。**

**识别规则：**

| 识别信号 | 权重 | 说明 |
|---------|------|------|
| 明确提及团队管理 | 🔴 强信号 | "我的团队"、"我带的人"、"下属"、"团队成员" |
| 管理职位/职级 | 🔴 强信号 | 总监、经理、组长、leader、TL、GM、主管 |
| 团队规模描述 | 🔴 强信号 | "团队有X人"、"管理X个人"、"N个直属" |
| 资源分配决策 | 🟡 中信号 | "HC分配"、"预算管理"、"资源调配" |
| 绩效评估权限 | 🟡 中信号 | "给他打绩效"、"团队考核"、"3.25/3.5" |
| 跨团队协调 | 🟡 中信号 | "跟XX部门协调"、"对齐目标"、"协同" |
| 人才培养提及 | 🟡 中信号 | "培养人"、"带新人"、"团队梯队" |
| 战略/方向决策 | 🟢 弱信号 | "业务方向"、"战略规划"、"技术选型" |

**判定规则（决策表）：**

| 条件 | 判定结果 | 说明 |
|------|---------|------|
| 任一🔴强信号命中 | → `is_manager = True` | 强信号权重3，单个即可判定 |
| 🟡中信号 ≥ 2个 且 置信度 > 0.4 | → `is_manager = True` | 中信号权重2，需组合判定 |
| 其他情况 | → `is_manager = False` | 默认非管理干部 |

**置信度计算**：`confidence = min(加权得分 / 最大可能得分 × 2, 1.0)`（加权：🔴×3 + 🟡×2 + 🟢×1）

**输出信息**：`is_manager`（布尔）、`confidence`（0-1）、`signals`（命中的信号列表）、`team_info`（团队规模/管理层级，仅管理干部）

**判定结果影响 Step 8：**
- `is_manager = True` → Step 8 生成「管理干部画像」（基础画像 + 多场景管理表现）
- `is_manager = False` → Step 8 生成「普通员工画像」（含战场画像分析，沿用原有逻辑）

#### ⭐⭐ 场景类型对 Step 8 的特殊控制（v2.8.0 新增）

> **核心原则：Step 1 识别的场景类型不仅决定方法论选择，还直接控制 Step 8 的画像生成策略。**

| 场景类型 | Step 8 行为 | 说明 |
|---------|------------|------|
| **离职访谈** | ⏭️ **跳过 Step 8** | 离职场景重点在离职原因分析，员工画像价值有限 |
| **招聘面试** | 🔄 **切换为「候选人综合画像」** | 不使用员工/管理干部画像，改为候选人综合画像（含岗位匹配+STAR场景） |
| **其他场景** | ✅ **沿用双轨画像逻辑** | 根据 `is_manager` 判定生成管理干部画像或普通员工画像 |

> ⚠️ **招聘面试场景前置要求**：用户在开始时应提供候选人应聘的**岗位职责**和**岗位要求**，用于 Step 8 的岗位匹配分析。如未提供，在 Step 4 展示分析框架时提醒用户补充。

#### ⭐⭐ 内容复杂度评估与快速模式（v2.12.0 新增）

> **核心原则：在 Step 1 完成场景识别后，自动评估访谈内容的复杂度。对于简单/短访谈记录，自动启用「快速分析模式」，显著减少交互轮次和分析时间。用户也可手动请求快速模式。**

**复杂度评估维度（0-10分，≤3分自动启用快速模式）：**

| 维度 | 0分 | 1分 | 2分 |
|------|-----|-----|-----|
| **字数** | ≤1500字 | 1500-5000字 | >5000字 |
| **话题数量** | ≤3个 | 3-5个 | >5个 |
| **访谈人数** | 1人 | 2人 | >2人 |
| **场景复杂度** | 入职/转岗等 | 绩效/晋升 | 组织诊断/综合分析 |
| **矛盾/冲突信号** | ≤1个 | 2-3个 | >3个 |

**快速模式判定**：`总分 ≤ 3 → 自动启用快速模式`（跳过 Step 4/6/9/10/11/13）

> ⚠️ 组织诊断和综合分析场景复杂度维度至少+2分，因此**不允许**进入快速模式

**快速模式行为变更：**

| 步骤 | 标准模式 | 快速模式 | 说明 |
|------|---------|---------|------|
| Step 1 | ✅ 完整执行 | ✅ 完整执行 | 场景识别+管理干部判定+复杂度评估 |
| Step 2 | ✅ 完整执行 | ✅ 完整执行 | 文本修正不可省略 |
| Step 3 | ✅ 完整执行 | ✅ 完整执行 | 方法论匹配（快速模式强制1主+1辅=2种） |
| **Step 4** | ✅ 展示框架请确认 | ⏭️ **跳过确认** | 自动采用推荐框架，在报告开头附注 |
| Step 5 | ✅ 完整分析 | ✅ 完整执行 | 核心分析不可省略 |
| **Step 6** | ✅ 交叉验证 | ⏭️ **简化执行** | 仅做关键矛盾检测，跳过一致性详细验证 |
| Step 7 | ✅ 洞察生成 | ✅ 完整执行 | 核心输出不可省略 |
| Step 8 | ✅ 画像生成 | ✅ 完整执行 | 非离职场景仍强制执行 |
| **Step 9** | ✅ 组织画像 | ⏭️ **跳过** | 快速模式不触发组织画像（组织诊断场景不允许进入快速模式，故此步骤必然不适用） |
| **Step 10** | ⭐ 强制询问 | ⏭️ **默认跳过** | 不询问，直接跳过可视化 |
| **Step 11** | ⭐ 强制询问 | ⏭️ **默认跳过** | 不询问，直接跳过结论文章 |
| Step 12 | ✅ 输出报告 | ✅ **简化检查** | 仅检查 Step 1/2/3/5/7/8，跳过项不纳入 |
| **Step 13** | ⭐ 强制询问 | ⏭️ **默认跳过** | 不询问，直接跳过腾讯文档 |

> ⚠️ **快速模式约束**：
> - 用户可随时通过指令 `"切换为完整模式"` 或 `"详细分析"` 退出快速模式
> - 组织诊断和综合分析场景 **不允许** 进入快速模式（complexity_score 必然 > 3）
> - **场景识别置信度 < 0.7 时禁止进入快速模式**（低置信度需用户确认场景，应走标准模式）
> - 快速模式下 Step 4 框架展示改为在报告开头附注：`⚡ 本次使用快速分析模式（复杂度评分：X/10），如需完整分析请回复"详细分析"`
> - 快速模式预估可减少 **40-50% 的交互轮次** 和 **30% 的 token 消耗**

### Step 2: 文本修正 ⭐⭐

> **核心原则：在开始正式分析前，先根据上下文修正访谈记录中的错别字和语句不通顺的地方，确保分析基于准确清晰的文本。**

#### 修正范围

**1. 错别字修正**
- 常见同音字错误（如：的/得/地、在/再、已/以等）
- 形近字错误
- 输入法选词错误

**2. 语句不通修正**
- 语序混乱的句子
- 缺少主语或宾语的句子
- 重复冗余的表达
- 标点符号错误

**3. 上下文理解修正**
- 基于上下文推断可能的正确用词
- 保留口语化表达，但修正明显错误
- 不改变原意，仅提升可读性

#### 修正规则

| 规则类型 | 示例 | 修正后 |
|---------|------|-------|
| **错别字** | "这个项目做的不错" | "这个项目做得不错" |
| **错别字** | "我已经以经准备好了" | "我已经已经准备好了" |
| **语句不通** | "因为很多问题，所以导致" | "因为有很多问题，所以导致" |
| **语句不通** | "他跟我说了很多，然后，然后..." | "他跟我说了很多，然后..." |
| **标点错误** | "我们要解决问题，然后才能前进" | "我们要解决问题，然后才能前进" |

#### 修正流程

**修正执行顺序：**

| 步骤 | 修正类型 | 规则 |
|------|---------|------|
| 1 | **错别字修正** | 基于常见错误字典（同音字/形近字/输入法选词错误） |
| 2 | **语句不通修正** | 基于语法规则（语序/缺少主宾语/冗余表达） |
| 3 | **标点符号修正** | 标点规范化 |
| 4 | **上下文一致性检查** | 确保修正后文本上下文连贯 |

**输出格式**：修正后文本 + 修正记录列表（每条含：原文片段、修正后片段、类型、理由）

#### 修正输出

在分析报告中，文本修正部分会以附录形式呈现：

```markdown
## 📝 文本修正记录（附录）

本次分析对访谈文本进行了以下修正：

| 类型 | 原文 | 修正后 | 理由 |
|------|------|-------|------|
| 错别字 | 做的不错 | 做得不错 | 地/得/用法规范 |
| 语句不通 | 因为很多问题，所以导致 | 因为有很多问题，所以导致 | 补充完整句子结构 |

**修正原则：**
- 不改变原意，仅提升可读性
- 保留口语化表达，仅修正明显错误
- 基于上下文推断，确保理解准确
```

#### 注意事项

- ⚠️ **保留原汁原味**：口语化表达不必过度修正，只需确保意思清楚即可
- ⚠️ **避免误判**：不确定的用词不要随意修改，保持原样
- ⚠️ **记录修正**：所有修正都要记录在报告中，方便追溯
- ✅ **提升准确性**：文本修正有助于提升后续方法论匹配的准确性

### Step 3: 方法论自动匹配引擎 ⭐

根据识别的场景，自动匹配最优方法论组合。

> **⭐⭐ 核心约束1（强制执行）：每个场景最多使用 3 种方法论（1个主方法论 + 最多2个辅助方法论），避免分析过于发散，确保深度优先。此约束不可违背，即使用户要求也必须遵守。**

> **⭐⭐ 核心约束1.5（v2.8.0 新增，优先执行）：优先选用 2 种方法论（1个主方法论 + 1个辅助方法论）。仅当访谈内容涉及多个复杂维度、单一辅助方法论无法覆盖关键分析需求时，才启用第3种方法论。默认应以 1主+1辅 为标准配置。**

> **⭐⭐ 核心约束2（强制执行）：组织诊断类方法论（杨三角理论 M1 和韦斯伯德六盒模型 M2）在分析中最多只选择一个，且优先使用杨三角理论（M1）。如果初始映射中包含六盒模型（M2），自动替换为杨三角理论（M1）。**

#### 方法论匹配算法

**方法论匹配规则（决策表）：**

| 规则优先级 | 规则 | 说明 |
|-----------|------|------|
| P0 | **总数硬限制**：≤ 3种（1主+最多2辅） | ⭐ 强制约束，不可违背 |
| P0 | **组织诊断互斥**：M1和M2最多选一个，优先M1 | 若同时出现，保留M1移除M2 |
| P1 | **默认优先2种**（1主+1辅） | v2.9.0 默认配置 |
| P2 | **复杂度触发扩展**：内容复杂度 ≥ 0.7 时扩展至3种 | 仅高复杂度才增加第3种方法论 |
| P3 | **次要场景融入**：次要场景的主方法论可替换低优先级辅助方法论 | 不可超出总数限制 |
| P4 | **内容驱动微调**：检测到组织因素信号 → 追加M1（若未选用） | 替换优先级最低的辅助方法论 |

**匹配流程**：基础映射（场景→方法论表） → 次要场景融入 → 内容微调 → 组织诊断互斥检查 → 数量裁剪（按优先级排序，保留TopN） → AI增强维度激活（不计入方法论数量）

**输出信息**：`primary_methodology`（主方法论+理由+维度+权重）、`secondary_methodologies`（辅助方法论列表，≤2个）、`total_methodologies`（总数，≤3）、`ai_enhancement`（AI信号）、`analysis_strategy`（分析策略）

---

### M4: 冰山模型 - 胜任力评估分析

> **详细分析指南：** [iceberg-analysis.md](references/iceberg-analysis.md)

**核心框架：** 水面以上（知识+技能，20%可见）→ 水面以下（社会角色+自我认知+特质+动机，80%隐藏，决定长期绩效）

## 🤖 AI 时代增强分析维度（v2.0）

### 核心理念

> 分析访谈结果时，同步评估 **AI 因素**对访谈主题的影响——AI 既可能是问题的原因，也可能是解决方案的一部分。

### 五大 AI 分析维度

| 维度 | 分析焦点 | 与方法论的交叉分析 |
|------|---------|-----------------|
| 🛠️ **AI 工具素养** | 员工使用 AI 工具的广度/深度 | BEM:知识与技能 / 冰山:技能层 |
| 🔄 **人机协作** | 员工与 AI 协作的成熟度 | BEM:资源与工具 / 杨三角:员工能力 |
| 🔮 **岗位演进认知** | 员工对 AI 影响岗位的理解 | 推拉力:推力&拉力 / 职业锚:锚点变化 |
| 📈 **能力模型更新** | 员工的能力进化意愿和行动 | 冰山:隐藏层变化 / GROW:选项与意愿 |
| 🏢 **组织 AI 成熟度** | 组织 AI 支持环境的评估 | 杨三角:员工治理 / 韦斯伯德:支持机制 |

### AI 维度融入各场景的分析规则

| 分析场景 | 重点 AI 维度 | 分析视角 |
|---------|-------------|---------|

| **离职分析** | 🔮 + 🏢 | AI 带来的岗位焦虑是否为推力？组织 AI 支持不足是否为推力？ |
| **职业发展分析** | 📈 + 🔮 + 🔄 | AI 对职业锚的影响？AI 时代需要什么新能力？ |
| **组织诊断分析** | 🏢 + 🔄 + 📈 | 组织 AI 基建是否到位？AI 文化是否包容？ |
| **招聘面试分析** | 🛠️ + 🔄 + 📈 | 候选人的 AI 素养是否达标？人机协作能力如何？ |

## 🔬 分析方法论：假设驱动 + 矛盾检测 + 深度追因

### 假设驱动分析法

每次分析开始前，基于场景和方法论**先形成假设**，然后用访谈内容验证/推翻假设：

```
Step 1: 基于场景生成初始假设
   例（离职分析）:
   H1: 薪酬不满是主要推力
   H2: 职业发展受限是次要推力
   H3: 直属上级管理问题可能是隐性推力

Step 2: 用访谈证据验证每个假设
   H1: 🟢 支持（多次提及薪酬对比）
   H2: 🟡 部分支持（提到发展但情绪不强烈）
   H3: 🔴 推翻（对上级评价积极）

Step 3: 修正假设，形成新假设
   H4: 工作挑战不足可能是核心推力（新发现）

Step 4: 二次验证，形成最终结论
```

### 矛盾检测机制

自动检测访谈中的**自相矛盾**和**表里不一**：

| 矛盾类型 | 表现 | 分析策略 |
|---------|------|---------|
| **言行矛盾** | 说满意但语气消极 | 更信任情感信号 |
| **前后矛盾** | 前面说好后面说差 | 关注后半段（防御降低后更真实） |
| **程度矛盾** | 说"还好"但举了很多反例 | 关注事例而非总结性评价 |
| **归因矛盾** | 归因于外部但行为模式显示内因 | 综合分析行为证据 |
| **社会期望偏差** | 说"公司很好"但在找工作 | 识别社会期望回答 |

### 5 Whys 深度追因

对关键发现使用 5 Whys 追根溯源：

```
发现: 员工绩效下滑
  Why 1: 为什么绩效下滑？→ 项目交付质量下降
  Why 2: 为什么质量下降？→ 加班频繁导致粗心
  Why 3: 为什么加班频繁？→ 人手不足+需求膨胀
  Why 4: 为什么人手不足？→ 预算冻结+离职未补
  Why 5: 为什么预算冻结？→ 组织战略调整中
  
  根因：组织战略调整期的人力资源配置问题（治理归因）
  → 非个人能力问题（BEM模型验证：环境因素主导）
```

### 流程零：快速分析模式（v2.12.0 新增）⚡

```
触发方式 A（自动）：
  当 Step 1 复杂度评估得分 ≤ 3 时自动启用
  → 用户看到提示：⚡ 已自动启用快速分析模式（复杂度 X/10）

触发方式 B（手动）：
  用户: "快速分析这份访谈记录：[粘贴内容]"
  或: "简单分析一下"
  → 强制进入快速模式（跳过复杂度评估）

快速流程：
  → Step 1: 场景识别 + 复杂度评估
  → Step 2: 文本修正
  → Step 3: 方法论匹配（限制为2种）
  → Step 5: 多维度分析
  → Step 7: 洞察生成
  → Step 8: 画像生成（非离职场景强制）
  → Step 12: 输出报告（简化检查）

跳过：Step 4（框架确认）、Step 6（交叉验证简化）、Step 9/10/11/13（默认跳过不询问）
退出：用户可随时回复"详细分析"或"完整模式"切换回标准13步流程
```

### 流程一：自动分析模式（推荐）

```
用户: "帮我分析这份访谈记录：[粘贴访谈内容]"
（也支持提供腾讯文档链接）

→ Step 1-4: 场景识别 → 文本修正 → 方法论匹配 → 展示框架（请用户确认）
→ Step 5-7: 多维度分析 → 交叉验证 → 洞察生成
→ Step 8:   画像生成（⚠️ 非离职场景强制执行）
→ Step 9:   组织画像（仅组织诊断场景）
→ Step 10/11/13: 批量询问可选步骤（可视化/结论文章/腾讯文档）
→ Step 12:  输出完整报告

⭐⭐⭐ 输出前检查：逐项确认「强制完成检查清单」中所有步骤已执行或已询问。
```

### 流程二：用户指定场景

```
用户: "这是一份离职访谈记录：[粘贴访谈内容]"

```

### 流程三：用户指定方法论（≤3种）

```
用户: "用推拉力理论分析这份访谈记录：[粘贴访谈内容]"
→ Step 3 采用用户指定方法论（⚠️ 检查总数≤3种） → 后续同流程一
```

### 流程四：多文件综合分析

```
用户: "这5份访谈记录，帮我做综合分析：[粘贴内容]"
→ 逐份执行 Step 1-8（分别生成画像）→ 综合交叉分析 → 组织画像 → 后续同流程一
```

## 📚 方法论索引

本技能基于多种专业方法论，针对不同分析场景提供定制化分析框架：

### 🏛️ 组织诊断类

| 方法论 | 适用场景 | 参考文档 |
|-------|---------|---------|
| **杨三角理论** ⭐ | 组织健康度评估、快速诊断 | [yang-triangle-analysis.md](references/yang-triangle-analysis.md) |
| **用途** | 从员工思维、员工能力、员工治理三个支柱诊断组织能力 |
| **核心价值** | 简洁聚焦，源自中国企业实践，互联网公司优先使用 |

| **韦斯伯德六盒模型** | 组织全面诊断、系统性评估 | [weisbord-six-box-analysis.md](references/weisbord-six-box-analysis.md) |
| **用途** | 从目的、结构、关系、奖励、领导、支持机制六个维度诊断 |
| **核心价值** | 区分"系统性问题"与"个人抱怨"，更适合全面诊断 |

| 方法论 | 适用场景 | 参考文档 |

| **用途** | 从推力、拉力、留任力、移动障碍四维度分析离职决策 |
| **核心价值** | 系统分析离职驱动力，量化各力量维度影响 |
